# 面向环境-经济耦合研究的个人科研操作系统执行指南

- 日期：2026-04-16
- 状态：Draft v1
- 关联文档：[个人科研操作系统设计](./2026-04-16-personal-research-operating-system-design.md)
- 适用阶段：博士入学前准备期
- 文档类型：执行化设计

## 1. 文档用途

本文档用于把顶层设计转化为可以直接执行的工作流骨架。它重点回答四类问题：

1. 材料应该放在哪里
2. 每类对象应该如何命名与区分
3. 一项工作应如何从输入流转到输出
4. 入学前的三阶段任务应如何组织

本文档不追求工具细节最优解，而追求一套稳定、低摩擦、可长期演进的执行框架。

## 2. 执行原则

执行阶段继续遵守以下原则：

1. 目录优先于工具
   任何在线工具都可能变化，但本地目录与文件结构必须稳定

2. 单一对象单一归属
   同一类信息尽量只在一个系统中承担主职责，避免重复维护

3. 先最小可用，再逐步扩展
   先把流程跑通，再考虑插件、自动化与复杂联动

4. 每次工作都留下可追踪产物
   不允许只“看过”或“想过”而没有任何沉淀

5. AI 输出默认待核验
   AI 只提供草案和候选，不直接进入最终资产层

## 3. 系统角色分工

在执行层面，建议固定以下职责分工。

### 3.1 Zotero

唯一文献事实库，负责：

- 条目
- PDF
- 标签
- 分组
- 引用信息

不负责：

- 长篇研究思考
- 主题整合
- 方法理解沉淀

### 3.2 Obsidian

唯一研究认知库，负责：

- 文献笔记
- 方法卡片
- 问题笔记
- 数据卡片
- 研究备忘录

不负责：

- 文献主条目管理
- PDF 附件主存储

### 3.3 本地项目目录

唯一计算与输出工作区，负责：

- 数据
- 代码
- 分析结果
- 图表
- 阶段性文稿

### 3.4 AI 会话

临时协作层，负责：

- 初筛
- 对比
- 结构抽取
- 方法解释
- 代码辅助
- 草稿整理

不负责：

- 最终事实确认
- 最终结论定稿

## 4. 本地目录结构

建议你的科研主目录采用固定分层。这里不要求你一次性迁移所有旧材料，但后续新增内容应尽量遵守此结构。

```text
research/
├─ 00_inbox/
├─ 01_literature/
├─ 02_notes/
├─ 03_methods/
├─ 04_data/
├─ 05_projects/
├─ 06_outputs/
├─ 07_admin/
└─ 99_archive/
```

### 4.1 `00_inbox`

作用：临时入口。

放置：

- 临时下载文件
- 待整理截图
- 临时导出
- 尚未归类的零散材料

规则：

- 不长期存放
- 每周清空或转移

### 4.2 `01_literature`

作用：文献辅助目录，不替代 `Zotero`。

建议只放：

- 手工导出的综述清单
- 主题文献列表
- 阅读计划表
- 无法直接稳定放入 `Zotero` 的辅助材料

规则：

- 主 PDF 仍优先归 `Zotero`
- 这里更像文献管理的旁路工作台

### 4.3 `02_notes`

作用：`Obsidian` 所在目录或其核心同步目录。

建议子目录如下：

```text
02_notes/
├─ 00_dashboard/
├─ 01_literature_notes/
├─ 02_method_cards/
├─ 03_question_notes/
├─ 04_data_cards/
├─ 05_research_memos/
├─ 06_meeting_notes/
└─ 99_templates/
```

### 4.4 `03_methods`

作用：方法学习与最小示例目录。

建议子目录如下：

```text
03_methods/
├─ input_output/
│  ├─ concepts/
│  ├─ toy_examples/
│  ├─ scripts/
│  └─ references/
├─ gcam/
└─ ai_mechanistic/
```

当前阶段重心放在 `input_output/`。

### 4.5 `04_data`

作用：数据资产目录。

建议三层划分：

```text
04_data/
├─ raw/
├─ interim/
└─ processed/
```

规则：

- `raw/` 不手改
- 清洗结果进入 `interim/`
- 直接用于分析的整洁数据进入 `processed/`

### 4.6 `05_projects`

作用：围绕具体研究主题或练习课题建立独立项目。

建议一个项目一个目录，例如：

```text
05_projects/
├─ io-learning-mini/
├─ capital-formation-mapping/
└─ infrastructure-impact-scout/
```

项目目录建议统一结构：

```text
project-name/
├─ README.md
├─ data/
├─ notebooks/
├─ scripts/
├─ results/
└─ memo/
```

### 4.7 `06_outputs`

作用：对外可读输出。

放置：

- 周备忘录导出稿
- 月综述
- 主题报告
- 汇报图表

### 4.8 `07_admin`

作用：轻管理材料。

放置：

- 学习计划
- 时间安排
- 与导师方向有关的准备清单
- 任务总表

### 4.9 `99_archive`

作用：归档旧版本或不再活跃内容。

规则：

- 归档而不删除
- 不把当前活跃文件混入归档区

## 5. 命名规范

命名要服务于检索与复用，不追求花哨。

### 5.1 目录命名

采用：

`短横线小写英文`

例如：

- `io-learning-mini`
- `capital-formation-mapping`
- `weekly-review`

### 5.2 文档命名

建议按对象类型采用固定前缀。

#### 文献笔记

`lit-作者简写-年份-关键词.md`

示例：

`lit-miller-2021-mrio-overview.md`

#### 方法卡片

`method-主题名.md`

示例：

`method-leontief-inverse.md`

#### 问题笔记

`question-主题名.md`

示例：

`question-fixed-capital-and-emissions.md`

#### 数据卡片

`data-数据源名.md`

示例：

`data-exiobase.md`

#### 研究备忘录

`memo-YYYY-MM-DD-主题名.md`

示例：

`memo-2026-04-16-why-capital-formation-matters.md`

### 5.3 脚本命名

采用：

`动词_对象.py`

示例：

- `load_io_table.py`
- `build_leontief_matrix.py`
- `plot_emission_intensity.py`

## 6. Obsidian 中的对象设计

你的知识库需要控制对象类型，避免什么都写成“笔记”。

### 6.1 文献笔记

每篇文献笔记至少包含：

- 研究问题
- 研究对象
- 方法
- 数据
- 主要发现
- 局限
- 可借鉴点
- 与导师方向关系

这类笔记的重点不是复述全文，而是为后续比较和调用服务。

### 6.2 方法卡片

每张方法卡片至少包含：

- 这个方法解决什么问题
- 需要什么输入
- 核心矩阵或核心步骤是什么
- 输出结果如何解释
- 常见误用点是什么
- 与当前研究方向的连接点是什么

投入产出分析学习阶段，方法卡片应成为最高优先级资产。

### 6.3 问题笔记

每张问题笔记至少包含：

- 当前问题陈述
- 这个问题为什么重要
- 可能的机制链条
- 可能适用的方法
- 已知文献支持
- 当前不清楚的地方

问题笔记是选题池的前体，不要求一开始就写得很成熟。

### 6.4 数据卡片

每张数据卡片至少包含：

- 数据源名称
- 官方来源
- 空间范围
- 时间范围
- 主要变量
- 口径与限制
- 获取方式
- 适合回答什么问题

### 6.5 研究备忘录

研究备忘录是主题整合对象，至少包含：

- 主题
- 当前判断
- 支撑文献
- 可用方法
- 可用数据
- 下一步问题

与文献笔记相比，研究备忘录必须体现你的综合判断。

## 7. Zotero 组织建议

你已经有现成系统，因此这里不建议重构，只建议做增量规范。

### 7.1 最小组织维度

建议确保 `Zotero` 至少能支持以下三类视角：

1. 主题视角
   例如 `input-output`、`capital formation`、`infrastructure`、`resource use`

2. 类型视角
   例如 `review`、`method`、`dataset`、`application`

3. 相关度视角
   例如 `core`、`adjacent`、`background`

### 7.2 不建议做的事

- 不要在 `Zotero` 里写大量研究性长笔记
- 不要把 `Obsidian` 里的问题演化再复制回 `Zotero`
- 不要用过细标签导致维护成本过高

## 8. AI 会话使用规范

建议把 AI 会话按任务类型区分，而不是所有问题都丢进一个长线程。

### 8.1 四类核心会话

1. 文献检索会话
2. 文献阅读会话
3. 方法学习会话
4. 代码协作会话

### 8.2 每次会话的最小输入结构

建议每次提问至少明确：

- 任务目标
- 当前上下文
- 你需要的输出格式
- 哪些地方必须谨慎

例如：

```text
任务：帮我拆解这篇 IO 论文的方法结构
背景：我正在学习环境扩展投入产出分析
输出：按“问题-方法-数据-结果-局限”结构回答
要求：不要编造引文；不确定的地方明确说不确定
```

### 8.3 会话后处理

每次高价值 AI 会话结束后，只保留以下成果进入系统：

- 一条明确判断
- 一份结构化笔记草稿
- 一段可运行代码
- 一份待核验的文献列表

不要把完整聊天记录直接当成知识库。

## 9. 五个核心工作流

## 9.1 工作流一：文献发现与沉淀

标准路径：

`发现文献`
-> `进入 Zotero`
-> `初步标签`
-> `AI 初筛/结构抽取`
-> `人工核验`
-> `Obsidian 文献笔记`
-> `链接到方法卡片/问题笔记`

完成标准：

- 该文献在 `Zotero` 可被找到
- 该文献是否值得长期保留已有判断
- 高价值文献已进入 `Obsidian`

## 9.2 工作流二：投入产出分析学习

标准路径：

`选定一个概念或方法`
-> `阅读 1-3 篇相关材料`
-> `AI 辅助解释`
-> `构造最小示例`
-> `写方法卡片`
-> `记录仍不理解的问题`

完成标准：

- 能用自己的话解释它
- 能给出一个最小计算例子
- 能说明它适合回答什么问题

## 9.3 工作流三：数据源进入系统

标准路径：

`发现数据源`
-> `创建数据卡片`
-> `记录来源与口径`
-> `下载到 raw/`
-> `写初步读取脚本`
-> `判断是否值得纳入长期资产`

完成标准：

- 数据源基本信息完整
- 原始文件归档明确
- 有最小读取路径

## 9.4 工作流四：主题研究备忘录

标准路径：

`选定主题`
-> `汇总相关文献笔记`
-> `比较不同研究`
-> `提炼当前判断`
-> `列出方法与数据入口`
-> `形成备忘录`

完成标准：

- 不是文献堆砌
- 有明确“当前判断”
- 能直接服务于后续汇报或问题收敛

## 9.5 工作流五：每周回顾

建议每周固定一次，处理以下事项：

1. 清空 `00_inbox`
2. 整理本周新增文献
3. 检查是否新增方法卡片
4. 检查是否新增问题笔记
5. 输出一份周研究备忘录

每周回顾是系统维持长期稳定的关键动作。

## 10. 三阶段任务清单

这里不做周计划，只给阶段任务框架。

### 10.1 阶段一：系统搭建期

目标：让系统能跑。

必须完成：

1. 固定主目录结构
2. 固定 `Obsidian` 的五类核心对象
3. 明确 `Zotero`、`Obsidian`、本地项目目录的边界
4. 跑通一篇文献的完整流转
5. 跑通一张方法卡片的完整流转
6. 建立每周回顾习惯

阶段通过标准：

- 你知道每类材料该放哪里
- 你能稳定完成一次“文献 -> 笔记 -> 备忘录片段”的转化

### 10.2 阶段二：方法主线训练期

目标：让投入产出分析成为你的可用方法。

必须完成：

1. 梳理 IO 方法学习顺序
2. 为至少 5 个核心概念建立方法卡片
3. 跑通至少 1 个最小 IO 示例
4. 拆解至少 5 篇 IO 应用论文
5. 建立一个 `input_output/` 方法目录
6. 形成至少 1 份 IO 学习备忘录

阶段通过标准：

- 你能读懂普通 IO 方法部分
- 你能解释核心概念
- 你能用代码跑通基础流程

### 10.3 阶段三：研究整合期

目标：让系统开始对接导师方向。

必须完成：

1. 形成导师方向主题地图
2. 建立至少 3 个问题笔记
3. 建立至少 3 张关键数据卡片
4. 形成至少 2 份主题研究备忘录
5. 搭建至少 1 个与导师方向相关的小型项目目录

阶段通过标准：

- 你能说清楚当前准备和导师方向之间的连接
- 你能展示一套初步但完整的研究资产系统

## 11. 模板最小集合

后续如果继续脚手架化，建议优先落以下模板。

1. 文献笔记模板
2. 方法卡片模板
3. 问题笔记模板
4. 数据卡片模板
5. 研究备忘录模板
6. 周回顾模板
7. 项目 README 模板

在当前阶段，不建议优先制作大量复杂模板。先把这七个模板做稳即可。

## 12. 第一批应优先创建的对象

如果你现在开始搭系统，建议优先创建而不是继续抽象讨论的对象有：

1. 一个 `Obsidian` 根目录结构
2. 一张 `method-leontief-inverse.md`
3. 一张 `question-fixed-capital-and-emissions.md`
4. 一张 `data-<核心数据库>.md`
5. 一份 `memo-YYYY-MM-DD-why-io-first.md`
6. 一个 `05_projects/io-learning-mini/` 项目目录

这 6 个对象足以作为整个系统的第一批支点。

## 13. 下一步实施顺序

建议后续按下面顺序继续：

1. 先创建目录结构
2. 再创建模板文件
3. 再创建第一批核心对象
4. 再把文献、方法和备忘录工作流跑一遍
5. 最后再决定是否加插件、自动化或脚本联动

## 14. 当前版本边界

本版本已足以指导你开始搭建工作流，但仍有三部分未细化：

1. `Obsidian` 模板文件的具体字段
2. 你的本地科研目录的实际创建
3. 第一批方法卡片与研究备忘录的实际填充

这些可以在下一步直接落到文件和目录中，而不必继续停留在概念层。
